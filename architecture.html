ServiceOps — System Architecture

v0.1 baseline 2026-06-08 Status: locked for MVP

Multi-tenant operations platform for service agencies in East Africa. Same code, same database, different tenant configuration — runs a 5-cleaner agency in Nyali and a 200-staff caregiving firm in Nairobi.

0. Vision & Constraints

What we are building

ServiceOps is a multi-tenant operations platform for service agencies. Day-one verticals:

  • Domestic cleaning (one-off, weekly, deep clean)
  • House-help placement (interview → match → payroll)
  • Laundry pickup / delivery
  • Caregiving (elderly, post-natal, child)
  • Pest control
  • Property maintenance

The platform itself is the product. Tenant configuration does the rest.

What we are NOT building (yet)

Deferred post-MVP: consumer marketplace search, full CRM pipelines, native mobile apps, Cal.com / Novu / Budibase / NocoDB / Invoice Ninja / Keycloak / Twenty / n8n / OSRM / PostHog / Directus integrations.

Architectural constitution (the 10 rules)

  1. Own the business model. Users, bookings, jobs, payments live in our DB. External systems are replaceable adapters.
  2. Treat every external call as eventually-delivered or never-delivered. Outbox pattern. No synchronous coupling.
  3. Multi-tenant data model from line 1. Enforce via RLS later. Don't retrofit.
  4. Money is integer minor units. Always. No floats, no decimals in code.
  5. No DELETE in production. Soft delete + PII redaction.
  6. Every state transition is auditable. Actor, from, to, at, reason.
  7. Idempotency keys on every mutation. Not just payments.
  8. Offline-first for workers. Local queue, sync on reconnect, conflict resolution.
  9. Job types are an enum, not separate tables. ServiceOps scales by adding enum values + service config.
  10. Single Next.js codebase. Route groups for personas, not separate apps.

3.2 Critical constraints

-- Concurrency: no two CONFIRMED/IN_PROGRESS bookings for the same worker
ALTER TABLE bookings ADD CONSTRAINT no_worker_double_book
  EXCLUDE USING gist (
    worker_id WITH =,
    tstzrange(scheduled_start, scheduled_end) WITH &&
  ) WHERE (status IN ('CONFIRMED', 'IN_PROGRESS'));

-- Idempotency: M-Pesa callbacks
ALTER TABLE payments ADD CONSTRAINT uniq_mpesa_callback
  UNIQUE (merchant_request_id, checkout_request_id);

-- Outbox: at-least-once delivery
ALTER TABLE outbox ADD CONSTRAINT uniq_outbox_event
  UNIQUE (event_type, event_key);

-- Optimistic concurrency
ALTER TABLE bookings ADD COLUMN version INTEGER NOT NULL DEFAULT 1;

3.3 Multi-tenancy strategy

Rule: tenant_id UUID NOT NULL on every domain table from migration #1. Enforce via Postgres Row-Level Security. Even single-tenant on day 1. 
ALTER TABLE bookings ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_bookings ON bookings
  USING (tenant_id = current_setting('app.tenant_id', true)::uuid);

ALTER TABLE bookings FORCE ROW LEVEL SECURITY;

API middleware extracts tenant_id from the JWT, then sets the session GUC for RLS. Every query path has a "cross-tenant access denied" test.

7. Cross-Cutting Concerns

7.1 Outbox pattern

Rule: every external call is initiated from a row in outbox, written in the same transaction as the business state change. A worker drains the outbox. 
CREATE TABLE outbox (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL,
  channel         TEXT NOT NULL,        -- 'whatsapp' | 'sms' | 'email' | 'mpesa_stk' | 'mpesa_b2c'
  event_type      TEXT NOT NULL,
  event_key       TEXT NOT NULL,        -- idempotency key
  payload         JSONB NOT NULL,
  status          TEXT NOT NULL DEFAULT 'pending',
  attempts        INT  NOT NULL DEFAULT 0,
  next_attempt_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  delivered_at    TIMESTAMPTZ,
  last_error      TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX uniq_outbox_event ON outbox (event_type, event_key);

Drain worker uses FOR UPDATE SKIP LOCKED to claim batches, dispatches via WhatsApp / M-Pesa / SMS, marks delivered or schedules retry with backoff (1s, 5s, 30s, 5m, 30m). After max_attempts, marks DEAD and alerts ops.

7.2 Idempotency

Every mutation endpoint accepts Idempotency-Key: <uuid>. Server caches the response for 24h. Not just payments — every POST/PUT/PATCH.

7.3 Money

  • All money as BIGINT minor units in DB. Never Number in code.
  • API accepts money as { minor: "2500", currency: "KES" }. Never decimals.
  • Separate amount_gross, amount_fee, amount_net columns.
  • Ledger table separate from payments. payments = gateway record; ledger = authoritative double-entry view.

7.4 Soft delete + PII redaction

Prisma middleware injects WHERE deleted_at IS NULL on every read. No DELETE in production. On Kenya DPA erasure request: scrub name, phone, email to REDACTED-<id> but keep the row for accounting.

7.5 Audit log

Every state transition writes a job_event row in the same transaction. Append-only. Partitioned monthly.

13. Glossary

TermMeaning
TenantA customer of ServiceOps — an agency. Owns users, services, workers, bookings.
ServiceA bookable offering (e.g. "2-bedroom deep clean"). Has a job_type enum value.
Job typeThe category of work: CLEANING, LAUNDRY, PLACEMENT, CAREGIVER, AIRBNB, PEST_CONTROL, MAINTENANCE.
BookingA customer's intent to purchase a service at a time. Has FSM. Becomes a Job when assigned.
JobThe actual work. Has FSM. Belongs to a Worker.
CandidateA house-help or caregiver applicant, pre-placement.
PlacementA successful match: candidate → customer, possibly realized as a recurring Job.
OutboxA row in outbox table representing a pending external call. Drained by worker.
FSMFinite State Machine. The allowed transitions for a Booking / Job / Payment.
Idempotency keyA client-supplied UUID that lets the server recognize a repeated request.
LipanaM-Pesa-as-a-service. Wraps Daraja with Stripe-like webhooks.
Reliability scorePer-worker number, 0-5, that drops on no-shows and recovers on completed jobs. Used in assignment.

End of v0.1 architecture baseline. Next: API contract (OpenAPI), FSDs (M-Pesa, Worker offline, RLS), operational runbook, pilot onboarding checklist — pick whichever you want first.
